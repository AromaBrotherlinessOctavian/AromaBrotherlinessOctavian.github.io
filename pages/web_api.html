Web-API

Принцип работы Web-API

Web-API представляют собой такие же API, просто работающие с использованием Web-технологий и Web-протоколов, прежде всего HTTP.

Пользователь с неким IP-адресом посещает сайт, а сайт обращается к некому GeoIP API сервису, который сообщает серверу какой стране принадлежит данный IP-адрес.

Одна из основных техник в коммуникации - энкапсуляция (вложение): использование информации, описывающей и содержащей другую информацию. Например, в файле-архиве (.zip и т.д.), содержащем в себе другие файлы (а может даже и другие архивы), полезная информация заворачивается в некий вспомогательный "конверт" который обладает нужными свойствами (сжатие, шифрование, мета-информация и т.д.). И этот "конверт" можно завернуть в ещё один, другой (или тот же) "конверт". Протоколы могут выступать в роли носителя произвольных полезных нагрузок, которые могут представлять собой данные другого протокола/формата/языка.

Web-API обычно используют HTTP (впрочем, могут использовать и WebSocket) как средство передачи, а сама полезная нагрузка может быть закодирована в JSON или XML, а может быть и просто двоичными данными. Сам HTTP обычно заворачивается в TLS Record'ы, которые заворачиваются в TCP сегменты, которые заворачиваются в IP-датаграммы, которые заворачиваются в Ethernet-фреймы, которые потом кодируются радиоволнами или светом. Каждый уровень независим (но обычно они связаны в одну большую систему или несколько подсистем и могут в какой-то степени взаимодействовать) и у каждого уровня есть свой соответствующий компонент, который обрабатывает (или наоборот, создаёт) информацию данного уровня. И сама полезная нагрузка, будь то JSON, XML или двоичные данные, тоже обрабатывается соответствующим компонентом - имплементацией API, в соответствии с контрактом (самим API).

Различают SOAP Web-API (Web Services) и REST Web-API. В последнее время чаще встречаются REST API, которые более гибкие, не имеют состояния и имеют менее строгую структуру (впрочем, иногда строгость это хорошо), чем SOAP API.
